# 对话管理功能详细说明

**版本**: 1.0
**最后更新**: 2026-02-23

---

## 📋 功能概述

mPower_Rag 的对话管理功能支持多轮对话，让用户可以连续提问，系统会记住对话历史并保持上下文连贯性。

### 核心特性

- ✅ **多轮对话支持** - 自动维护对话历史，保持上下文连贯
- ✅ **对话管理** - 创建、切换、删除对话
- ✅ **历史记录** - 自动保存和加载对话历史
- ✅ **智能上下文** - 将对话历史格式化为 LLM 可理解的格式
- ✅ **会话隔离** - 不同对话之间的历史互不干扰


## 🎯 功能详解

### 1. 创建对话

**API 接口**:
```http
POST /api/v1/conversations
Content-Type: application/json

{
  "metadata": {
    "title": "车载测试咨询",
    "topic": "车载测试"
  }
}
```

**响应**:
```json
{
  "conversation_id": "conv_1708704000.123",
  "message": "对话创建成功"
}
```

**前端操作**:
- 点击侧边栏 "➕ 新建对话" 按钮
- 自动生成对话 ID
- 自动设置为当前对话

---

### 2. 发送消息

**API 接口**:
```http
POST /api/v1/conversations/{conversation_id}/messages
Content-Type: application/json

{
  "question": "什么是车载测试？",
  "top_k": 5,
  "use_history": true
}
```

**响应**:
```json
{
  "answer": "车载测试是指...",
  "question": "什么是车载测试？",
  "source_documents": [...],
  "conversation_id": "conv_1708704000.123"
}
```

**处理流程**:
1. 验证对话是否存在
2. 加载对话历史
3. 格式化历史为上下文
4. 检索相关文档
5. 将上下文、问题、文档发送给 LLM
6. 生成答案
7. 保存用户消息和助手回复到对话历史
8. 返回答案和源文档

---

### 3. 多轮对话

**工作原理**:

对话历史会自动格式化为 LLM 可理解的格式：

```
以下是对话历史：

用户: 什么是车载测试？
助手: 车载测试是指...

用户: 测试流程有哪些步骤？
助手: 车载测试的流程包括...

当前问题：如何准备测试报告？
```

**优势**:
- LLM 可以参考之前的对话
- 答案更连贯、更准确
- 无需用户重复提供上下文

**示例**:

```
第 1 轮：
你：什么是车载测试？
助手：车载测试是指对车辆及其组件进行的各种测试...

第 2 轮：
你：测试流程有哪些步骤？
助手：根据前面提到的车载测试，测试流程包括以下步骤...

第 3 轮：
你：需要注意什么？
助手：在进行上述测试流程时，需要注意以下几点...
```

---

### 4. 列出对话

**API 接口**:
```http
GET /api/v1/conversations?limit=20
```

**响应**:
```json
{
  "conversations": [
    {
      "id": "conv_1708704000.123",
      "created_at": "2024-02-23T08:00:00",
      "updated_at": "2024-02-23T09:00:00",
      "message_count": 5,
      "metadata": {...},
      "messages": [...]
    },
    ...
  ],
  "total": 10
}
```

**前端显示**:
- 侧边栏显示所有对话
- 每个对话显示 ID 和消息数量
- 按更新时间倒序排列

---

### 5. 删除对话

**API 接口**:
```http
DELETE /api/v1/conversations/{conversation_id}
```

**响应**:
```json
{
  "message": "对话删除成功"
}
```

**前端操作**:
- 点击对话旁边的 🗑️ 按钮
- 确认删除
- 如果删除的是当前对话，自动切换到其他对话或清空

## ⚙️ 配置选项

### 对话管理器配置

```python
# src/core/conversation.py
conversation_manager = ConversationManager(
    max_history=10  # 最大历史记录数量
)
```

**说明**:
- `max_history`: 保留的历史消息数量
- 当超过限制时，自动清理旧消息
- 系统消息（role="system"）不受此限制

### RAG 引擎配置

```python
# src/core/rag_engine.py
rag_engine = RAGEngine(
    conversation_manager=conversation_manager,  # 对话管理器
    llm_provider="deepseek",
    llm_model="deepseek-chat",
    llm_api_key="sk-...",
    temperature=0.7,
    max_tokens=2000
)
```

### API 请求配置

```python
{
  "question": "用户问题",
  "top_k": 5,              # 返回文档数量
  "use_history": true      # 是否使用对话历史
}
```

2. **上下文长度**

   - 考虑添加对话缓存
   - 减少重复的数据库查询

## 🔒 安全考虑

### 对话隔离

- 每个对话的互不影响
- 删除对话会清除所有相关数据
- 对话 ID 是唯一的

### 数据隐私

- 对话历史存储在内存中（当前实现）
- 重启服务后数据会丢失
- 如需持久化，需要集成数据库

### API 安全

- 生产环境需要添加认证
- 限制对话创建频率
- 验证对话 ID 的有效性

## 🆘 常见问题

### Q: 对话历史会自动清理吗？
A: 会。当消息数量超过 `max_history` 时，旧消息会被自动清理。

### Q: 可以手动清空对话历史吗？
A: 目前不支持，但可以删除对话并创建新对话。

### Q: 多个用户可以共享对话吗？
A: 目前不支持。每个对话是独立的，多用户共享需要额外开发。

### Q: 对话数据会持久化吗？
A: 目前不会。对话存储在内存中，重启服务后数据会丢失。

### Q: 如何限制对话的创建频率？
A: 需要在 API 层添加限流机制（如 FastAPI 的 `slowapi`）。
